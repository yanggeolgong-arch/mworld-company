## 1. 초등학생도 이해할 수 있는 요약

### 🎯 왜 느리다고 느껴지나요?

**비유로 설명하면:**

1. **선물 상자 문제**  
   우리 사이트는 "빈 상자"를 먼저 보여줍니다.  
   진짜 내용물(맛집 사진, 글)은 나중에 JavaScript가 실행된 뒤에야 화면에 나타납니다.  
   그래서 처음에는 빈 화면처럼 보이다가, 7초쯤 지나서야 큰 사진이 뜹니다.

2. **사진이 너무 큼**  
   화면에 보이는 크기보다 훨씬 큰 사진을 다운로드합니다.  
   예: 200x150 픽셀로 보이는데, 800x600 크기로 받아옴 → 2.3MB 정도 낭비.

3. **"나중에 로드" 설정**  
   가장 중요한 큰 사진에 "나중에 로드해"라고 되어 있어서,  
   브라우저가 그 사진을 늦게 불러옵니다.

4. **폰트 10개**  
   글자 모양(폰트)을 10개나 불러와서,  
   그만큼 더 기다려야 합니다.

## 3. LCP가 느린 이유 (상세)

### 3-1. LCP 요소가 무엇인가

- **위치**: 화면 아래쪽 "나머지 7개" 그리드의 **자매국수** 이미지 (6.avif)
- **선택자**: `div.grid > button.bg-[#fafaf5] > div.relative > img`

### 3-2. LCP Discovery 검사 결과 (모두 실패)

| 검사 항목 | 결과 | 의미 |
|-----------|------|------|
| `fetchpriority=high` 적용 | ❌ | LCP 이미지에 우선 로드 힌트가 없음 |
| 초기 HTML에서 요청 발견 | ❌ | **처음 HTML에는 이 이미지가 없음** (JS로 나중에 그려짐) |
| lazy 로드 미적용 | ❌ | **LCP 이미지에 `loading="lazy"`가 적용됨** → 더 늦게 로드됨 |

### 3-3. LCP 시간 구성 (추정)

1. **HTML 로드** (~75ms) – 빠름  
2. **JavaScript 로드 + 실행** – Next.js, React 번들  
3. **React 렌더링** – `shops`가 `useEffect` 이후에 채워짐  
4. **DOM 업데이트** – 그 후에야 이미지 태그가 화면에 생김  
5. **이미지 로드** – `loading="lazy"` 때문에 viewport에 가까워질 때까지 대기  
6. **렌더링 완료** – 여기서 LCP 7.2초 기록

## 4. 개선이 잘 안 되는 구조적 이유

### 4-1. 클라이언트 전용 렌더링 (CSR)

```tsx
// JejuGourmetBest10.tsx
const [shops, setShops] = useState<Shop[]>([]);  // 처음엔 빈 배열!

useEffect(() => {
  setShops([...initialShops].sort(() => Math.random() - 0.5));
}, []);  // JS 실행 후에야 데이터가 채워짐
```

- 처음 HTML에는 맛집 목록과 이미지가 **없음**
- JavaScript가 실행되고 `useEffect`가 돌아간 뒤에야 `shops`가 채워지고, 그때 이미지가 DOM에 추가됨
- 그래서 LCP 이미지가 **초기 HTML에서 발견 불가** → LCP가 크게 늦어짐

### 4-2. 랜덤 셔플 때문에 SSR이 어려움

- `sort(() => Math.random() - 0.5)`로 매번 순서를 바꿈
- 이걸 서버에서 하면, 사용자마다 다른 HTML이 나가서 캐시 이점이 줄어듦
- 그래서 현재는 클라이언트에서만 렌더링하는 구조

### 4-3. LCP 후보가 잘못된 이미지

- LCP로 잡힌 이미지는 **위쪽 캐러셀**이 아니라 **아래쪽 그리드**의 이미지
- 아래쪽 이미지는 `loading="lazy"`로 설정되어 있어, viewport에 들어올 때까지 로드를 미룸
- 결과적으로 LCP 시간이 더 길어짐

### 4-4. 이미지 최적화 부족

- **예상 절감량**: 2,344 KiB
- **LCP 개선 가능**: 약 2.95초
- 원인:
  - 압축률 부족
  - 표시 크기(예: 200x150)보다 훨씬 큰 해상도(예: 808x606)로 제공

## 5. 개선 방향 (우선순위)

### 🔴 1순위: LCP 이미지의 lazy 로드 제거

- "나머지 7개" 그리드 중 **첫 화면에 보이는 이미지**에는 `loading="lazy"`를 제거
- 또는 LCP 후보가 될 수 있는 이미지에만 `loading="eager"` 적용

### 🔴 2순위: 초기 HTML에 콘텐츠 포함 (SSR/SSG)

- 최소한 **첫 화면 3~6개**는 서버에서 HTML에 포함
- 랜덤 순서는 클라이언트에서만 "보여주는 순서"를 바꾸고,  
  실제 데이터는 서버에서 미리 렌더링

### 🟠 3순위: 이미지 최적화

- Next.js Image `sizes`를 실제 표시 크기에 맞게 조정
- `quality` 조정으로 용량 절감 (예: 75 → 65~70)
- 필요 시 WebP/AVIF 포맷 유지하면서 해상도 축소

### 🟡 4순위: 폰트 로딩

- Pretendard 279KB, 10개 요청 → 필요한 weight만 로드
- `font-display: swap` 적용으로 텍스트가 먼저 보이게

### 🟢 5순위: OSM iframe

- 상세 모달 안의 지도 iframe은 모달이 열릴 때만 로드 (lazy)
- 초기 LCP에는 영향 최소화
